# 회고

## 잘된 점

- 예약 흐름(승객정보 입력 → 좌석배정 → 추가수하물 → 결제)을 세션 기반으로 누적시키고,
  결제 시점에 트랜잭션으로 일괄 처리하는 구조를 직접 설계하고 구현했습니다.
- 다중 승객 처리, 가는편/오는편 좌석 등급 분리 등 왕복 항공권의 다양한 케이스를
  고려해 구현했습니다.
- 예약 내역 조회에서 1:N JOIN으로 인한 행 증가와 LISTAGG 중복 문제를 직접 진단하고
  해결하면서, 실무적인 SQL 집계 패턴을 익혔습니다. (자세한 내용은
  [TROUBLESHOOTING.md](TROUBLESHOOTING.md))

## 아쉬운 점 / 다음에 적용하고 싶은 것

- 사전에 확정되지 않은 부분이 생각보다 많아서, 개발 중 방향을 자주 다시 맞춰야
  했습니다. 다음에는 초기 기획/명세 단계를 더 구체적으로 세워야겠다고 느꼈습니다.
- 결제 흐름에서 클라이언트 입력값을 그대로 신뢰하는 부분이 있었고, 그중 일부
  (수하물 가격 재계산)는 발표 이후 개선했습니다. ([IMPROVEMENTS.md](IMPROVEMENTS.md))
  다만 수하물 선택 단계 자체의 수량 검증 등 추가로 개선할 부분이 남아있습니다.
- 결제 단계는 예약 정보를 DB에 저장하는 부분까지 구현했고, 실제 결제 대행(PG) 연동,
  비밀번호 암호화, 배포는 MVP 범위에서 제외했습니다.

## 협업

- Git 브랜치 전략(feature/fix)을 기반으로 79건의 커밋과 PR 병합을 통해 5인 팀과
  협업했습니다. 코드 충돌 해결, 기능 단위 커밋, 기획명세서 작성을 함께 진행하며
  팀 프로젝트에서의 협업 방식을 익혔습니다.
